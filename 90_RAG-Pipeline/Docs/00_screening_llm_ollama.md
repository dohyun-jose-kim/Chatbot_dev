# LLM 스크리닝에 Ollama를 선택한 이유

## 배경: 왜 LLM 스크리닝이 필요한가

키워드 스크리닝(Phase 1)은 PART/CATEGORY/FN 3개 그룹의 키워드가 abstract에 존재하는지 기계적으로 확인한다. 6,584편 중 5,590편(84.9%)이 통과했다.

문제는 키워드 매칭의 한계다:

- "fish"와 "collagen"이 있으면 통과하지만, 실제로는 수산부산물이 아닌 양식 사료 연구일 수 있다
- "antioxidant"와 "shrimp"가 있어도 새우 양식 환경 연구일 수 있다
- 키워드가 맥락 없이 매칭되므로 주제와 동떨어진 논문이 포함된다

따라서 LLM이 abstract를 **읽고** 맥락을 이해한 뒤, "이 논문이 수산부산물 유래 기능성 물질 연구인가?"를 판정하는 단계가 필요하다.

## 선택지 비교

| 기준 | 클라우드 API (Claude, OpenAI, Gemini) | 로컬 LLM (Ollama) |
|------|--------------------------------------|-------------------|
| 비용 | 유료 ($1~5 / 5,590편) | 무료 |
| 속도 | 빠름 (병렬 요청 가능) | 느림 (로컬 GPU 의존) |
| 품질 | 높음 (대형 모델) | 충분 (YES/NO 판정 수준) |
| 설정 | API 키 발급 필요 | `brew install` + 모델 다운로드 |
| 데이터 | 외부 서버로 전송됨 | 로컬에서만 처리 |
| 인터넷 | 필요 | 불필요 |

## 왜 Ollama인가

1. **비용 0원**: 논문 스크리닝은 반복 실험이 필요하다. 프롬프트를 수정하면서 여러 번 돌려봐야 하는데, 클라우드 API는 매번 비용이 발생한다.

2. **작업 난이도가 낮다**: abstract를 읽고 YES/NO를 판정하는 것은 LLM 입장에서 단순한 분류 작업이다. GPT-4 수준의 모델이 필요하지 않다. `gemma3:4b` 같은 4B 파라미터 경량 모델로 충분하다.

3. **프로토타이핑에 적합**: 로컬에서 바로 실행하고 결과를 확인할 수 있다. 프롬프트 튜닝 → 소량 테스트 → 전체 실행 사이클이 빠르다.

4. **향후 확장 가능**: Ollama로 검증한 프롬프트와 로직은 그대로 Claude API나 OpenAI API로 교체할 수 있다. 코드에서 `call_ollama()` 함수 내부만 바꾸면 된다.

## Ollama란

로컬 컴퓨터에서 오픈소스 LLM을 실행하는 도구다.

```
클라우드 API:                 Ollama:
내 코드 → 인터넷 → 서버       내 코드 → localhost:11434 → 로컬 모델
         (유료)                        (무료, 오프라인)
```

- 모델을 다운로드(`ollama pull`)하면 로컬에서 REST API로 호출 가능
- Mac의 경우 Metal(GPU 가속) 자동 활용
- Docker 없이 단일 바이너리로 동작

## 설치 및 실행

```bash
# 설치
brew install ollama

# 서버 시작 (별도 터미널에서 유지)
ollama serve

# 모델 다운로드 (~3GB)
ollama pull gemma3:4b

# 스크리닝 실행
python week_6/NLP_Advancement-week_4/01_screening/llm_screen.py
```

## 사용 모델: Gemma 3 시리즈

Google DeepMind의 Gemma 3. 숫자는 파라미터 수를 의미하며, 클수록 똑똑하지만 무겁다.

| 모델 | 파라미터 | 다운로드 크기 | 특징 |
|------|---------|-------------|------|
| **`gemma3:1b`** | **10억개** | **~1GB** | **가볍고 빠름 (현재 사용)** |
| `gemma3:4b` | 40억개 | ~3GB | 균형잡힌 선택 |
| `gemma3:12b` | 120억개 | ~8GB | 더 똑똑하지만 느리고 메모리 많이 씀 |

### 모델 변경 이력

처음에는 `gemma3:4b`를 선택했으나, Intel Mac(x86)에서 실행한 결과 **건당 1~2분**이 소요되었다.
5,590편 기준 약 140시간(5.8일)이 걸리는 계산이라 현실적으로 불가능했다.

Intel Mac은 Metal GPU 가속이 불가능하여 CPU만으로 추론하기 때문에 이런 속도 차이가 발생한다.
Apple Silicon(M1/M2) Mac에서는 Metal GPU를 활용하므로 같은 모델이라도 훨씬 빠르다.

따라서 `gemma3:1b`로 변경했다. YES/NO 분류 수준의 단순 작업에는 1B 모델로 충분하다고 판단했다.

그러나 `gemma3:1b`에서도 **건당 약 35초**가 소요되었다.
5,590편 기준 약 54시간(2.3일)으로, 여전히 Intel Mac에서는 현실적이지 않다.

| 모델 | 건당 소요 시간 | 5,590편 예상 총 시간 |
|------|-------------|-------------------|
| `gemma3:4b` | 1~2분 | ~140시간 (5.8일) |
| `gemma3:1b` | ~35초 | ~54시간 (2.3일) |

**결론: Intel Mac(x86)에서 로컬 LLM 스크리닝은 비현실적이다.**
Metal GPU 가속이 없는 환경에서는 경량 모델로도 대량 처리가 불가능하다.

### 향후 대안
1. **Apple Silicon Mac에서 실행** — M2 등에서는 Metal GPU 가속으로 속도가 수배 빨라짐
2. **클라우드 API 사용** — Claude/OpenAI/Gemini API는 병렬 요청이 가능하여 수십 분 내 처리 가능
3. **샘플 테스트 우선** — 소량(100편)으로 판정 품질을 먼저 확인한 뒤 전체 실행 방법을 결정
